# Проверка здоровья перед изменением программы

Это gate-скилл: он не пишет упражнения и не меняет `days.ts`/`workouts.ts` сам. Его задача — прежде чем любой другой скилл тронет программу, убедиться, что ограничения, на которых основано решение, ещё актуальны, и не переспрашивать то, что уже известно и не менялось.

Один и тот же человек и один и тот же паттерн — для домашней программы и зала. Health-check один на оба, не дублируется по программам.

## 1. Источники данных — в порядке приоритета

**Шаг 1а.** Прочитай `CLAUDE.md` целиком (разделы «Контекст пользователя» и «Правила работы») — это канонический текущий статус: диагнозы, активные ограничения, паттерн асимметрии.

**Шаг 1б.** Прочитай `health-checkins.md` в корне проекта — журнал прошлых запусков этого скилла (что спрашивали, что ответили, когда, что изменилось). Если файла нет — это первый запуск, создашь его в разделе 5.

**Шаг 1в.** Получи записи за последние ~14 дней из обеих таблиц Supabase (не весь массив — только свежее, здесь важна текущая картина, а не история):

## 2. Что считается «уже известным» и когда не переспрашивать

Для каждого отслеживаемого факта в `health-checkins.md` есть дата последнего подтверждения. Правило свежести:

- Если факт подтверждён **≤14 дней назад** и за это время в Supabase не появилось противоречащих сигналов (`knee_pain >= 1`, новый комментарий с упоминанием нового участка тела, `back_pain` рост) — считай факт актуальным, не спрашивай заново, просто отметь в выводе «без изменений с [дата]».
- Если факт старше 14 дней, или появились противоречащие сигналы — задай вопрос заново.
- **Исключение — шлюзовые факты, которые нельзя пропускать по свежести**, если предстоящее изменение программы прямо на них опирается: если `plan-next-phase`/`gym-plan-next-phase` собирается спроектировать этап, где вернётся осевая нагрузка на колено (присед/выпады/жим ногами полной амплитуды) — статус ортопедического осмотра переспрашивай всегда, даже если подтверждён недавно. Цена ошибки здесь выше, чем неудобство лишнего вопроса.

## 3. Предугадать вопросы — не общий чек-лист, а то, что реально может изменить план

Не спрашивай абстрактно «как самочувствие». Собери короткий список конкретных вопросов на основе разделов 1-2, обычно 2-4:

- **Статус колена у ортопеда** — если ограничение всё ещё активно в `CLAUDE.md` и (а) прошло ≥14 дней с последнего подтверждения, или (б) предстоящее изменение программы затрагивает нагрузку на колено — спроси прямо: диагноз снят / контрольный осмотр назначен на дату / без изменений.
- **Новые симптомы вне структурированных полей** — если в разделе 1в нашлись упоминания новых участков тела в комментариях, сформулируй вопрос про конкретный участок, а не общий.
- **Изменение условий** — сидячая работа, доступ к тренажёрам/гантелям, время на тренировку — спрашивай, только если прошло много времени с последнего подтверждения (например, >30 дней) или пользователь сам упомянул изменение образа жизни.
- **Точечное уточнение по тренду**, если Supabase показывает движение показателя, которое не объясняется существующими данными (например, всплеск `back_pain` без видимой причины в программе) — спроси, было ли что-то вне тренировок (сон, стресс, долгая поездка).

Задавай через `AskUserQuestion` **одним вызовом**, с конкретными вариантами ответа, один — рекомендуемый, если есть основание. Не включай в вызов факты, которые по разделу 2 признаны свежими.

Если по разделу 2 всё свежо и нет ни одного вопроса — не вызывай `AskUserQuestion` вообще, сразу переходи к разделу 4 с вердиктом 🟢.

## 4. Вердикт и советы по упражнениям

Сформируй короткий вывод (не полноценный отчёт — это отдаётся вызывающему скиллу, а не самостоятельный анализ):

```
## Health-check · [дата]

**Вердикт:** 🟢 без изменений / 🟡 уточнение, план не меняется по сути / 🔴 ограничения изменились

**Актуальные ограничения:** [колено, приоритет сторон, острая боль и т.д. — в 2-3 строки]

**Что учесть при выборе упражнений сейчас:** [1-3 конкретных совета — например: «колено всё ещё под защитой, не предлагай осевую нагрузку»; «утренняя боль в норме, наклоны стоя можно оставить»; «левая сторона по-прежнему приоритет»]
```

- 🟢 — ничего не изменилось, вызвавший скилл продолжает по своим правилам как обычно
- 🟡 — есть новая информация, но она не меняет активные ограничения принципиально (например, лёгкий новый дискомфорт без диагноза) — вызвавший скилл должен учесть это как caveat при выборе конкретных упражнений (через `exercise-review`/`gym-exercise-review`), но не обязан переделывать всю логику
- 🔴 — ограничения изменились по существу (диагноз снят/подтверждён иначе, новая серьёзная жалоба, осмотр перенёсся) — **CLAUDE.md должен быть обновлён до того, как вызвавший скилл продолжит**: сформулируй точную правку (что убрать/добавить в «Контекст пользователя» и «Правила работы»), покажи пользователю, дождись подтверждения, внеси правку в `CLAUDE.md`, и только затем верни управление вызвавшему скиллу с вердиктом 🔴→обработано

## 5. Обновить health-checkins.md

Допиши в конец файла (создай с этим заголовком, если его ещё нет):

```markdown
# Журнал health-check

## [дата]
- Вопросы: [что спросили]
- Ответы: [что ответили]
- Вердикт: 🟢/🟡/🔴
- Изменения в CLAUDE.md: [что именно, или "нет"]
- Вызвано из: [adjust-program / gym-adjust-program / plan-next-phase / gym-plan-next-phase / напрямую]
```

Этот файл не коммитится (см. `.gitignore`) — он локальная память между сессиями, а не часть кодовой базы.

## 6. Как другие скиллы должны использовать этот скилл

`adjust-program`, `gym-adjust-program`, `plan-next-phase`, `gym-plan-next-phase` вызывают `health-check` первым шагом, до сбора собственных данных. Если вердикт 🔴 и правка `CLAUDE.md` ещё не внесена — вызвавший скилл не продолжает. Если 🟢/🟡 — вызвавший скилл продолжает и учитывает советы из раздела 4 при выборе/замене упражнений.

`exercise-review`/`gym-exercise-review` не обязаны вызывать `health-check` для мелких правок текста (`how`/`cues`), но обязаны вызвать его, если причина правки — клиническая (новый симптом, снятие/подтверждение ограничения), а не просто улучшение формулировки.
