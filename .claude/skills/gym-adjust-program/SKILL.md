# Коррекция программы зала

## 1. Собрать данные

**Шаг 1а.** Если `gym-training-analysis.md` существует и создан не более 24 часов назад — прочитай его. Иначе — сначала выполни логику скилла `gym-analyze-results` (получи данные из Supabase, построй анализ), сохрани в `gym-training-analysis.md`, и только потом продолжай.

**Шаг 1в.** Прочитай текущие `src/data/gym/workouts.ts` и `src/data/gym/exercises.ts`.

**Шаг 1г.** Спроси у пользователя: "Какая последняя тренировка (workout_number), которую ты выполнил?" — корректируешь только тренировки после неё (незавершённые). Если ответ — последняя тренировка блока (сверь с максимальным `week`/номером тренировки в `src/data/gym/workouts.ts`) — блок уже пройден целиком: это не точечная коррекция, останови этот скилл и вызови `gym-plan-next-phase` вместо него.

## 2. Правила коррекции — по приоритету

Работаешь с тем же пользователем и тем же ограничением, что и в домашней программе: **тендинит связки надколенника слева, никакой осевой нагрузки через согнутое стоя колено**; средняя ягодичная и хамстринг слева — приоритет объёма (см. правила «Зал» в `CLAUDE.md`).

### Приоритет 1 — колено
Если `knee_pain >= 2` в одной из последних записей:
- Замени в ближайших незавершённых тренировках любые рискованные упражнения нижней части тела на safe-варианты из каталога (`gym_leg_curl_unilateral`, `gym_leg_extension` — с ещё более ограниченной амплитудой в `how`, `gym_hip_abduction_machine`, `gym_rdl_barbell`, `gym_glute_bridge_machine`, `gym_calf_raise`)
- Если рискованных упражнений в плане и так нет (весь каталог уже safe-only) — просто снизь вес на 15-20% и сократи амплитуду в затронутых упражнениях на ближайшую тренировку
- Если ни один safe-вариант из каталога не подходит и нужно добавить новое упражнение — вызови скилл `gym-exercise-review`, прежде чем писать новую запись в `src/data/gym/exercises.ts` (обязательные `target`/`equipment`, философия выбора тренажёра, чек-лист по колену)
- Это делается независимо от прочих правил ниже — колено всегда приоритет

### Приоритет 2 — стагнация (нет прогрессии 3+ тренировки при RPE ≤ «Норм»)
Если по конкретному `exercise_id` вес и повторы не растут 3+ тренировки подряд, а RPE в эти разы был «Легко» или «Норм» (не «Тяжело»/«Предел» — значит дело не в усталости):
- Увеличь нагрузку в ближайшей будущей тренировке с этим упражнением: вес +5-10% (округли до реалистичного шага: гантели ±2кг, штанга ±2.5-5кг, тренажёры ±2.5-5кг) либо, если сдвиг веса неудобен, подними верхнюю границу диапазона повторов на 2-3

### Приоритет 3 — перегруз (RPE «Предел» 2+ раза подряд по упражнению)
- Не повышай нагрузку в ближайшей тренировке с этим упражнением — либо оставь как есть, либо снизь вес на 5-10% на одну тренировку (разгрузка), затем возвращайся к прогрессии

### Если всё в норме
- Не меняй ничего в `workouts.ts`
- Сообщи пользователю: программа работает, коррекция не нужна, пуш не нужен

## 3. Как редактировать workouts.ts

- Корректируй только **незавершённые тренировки** (после той, что назвал пользователь в шаге 1г)
- Изменения вноси точечно: минимум правок, максимум эффекта — трогай только `sets`/`reps`/`weight` конкретного `exs[]`-элемента или, при замене упражнения по колену, сам `id`
- Не меняй `title`/`short`/`desc`, если это не связано с сутью правки

## 4. После изменений — коммит и пуш в main

1. Запиши изменения в `src/data/gym/workouts.ts` (и/или `src/data/gym/exercises.ts`, если добавляешь новый safe-вариант)
2. Добавь запись в конец `gym-training-analysis.md` (что изменено, почему — какой показатель триггернул)
3. Закоммить и запушить:

```bash
cd /путь/к/проекту  # используй реальный путь из workspace
git add src/data/gym/workouts.ts src/data/gym/exercises.ts gym-training-analysis.md
git commit -m "gym-adjust: [краткое описание что изменено и почему]"
git push origin main
```

Используй в commit message конкретику: какой показатель триггернул, что изменено. Например:
`gym-adjust: knee_pain=2 на тренировке 4 → leg_extension амплитуда снижена, RDL вес -10% на тренировках 5-6`

Версионирование service worker для этого пуша не требуется — сборка PWA идёт через vite-plugin-pwa с автоматическим кэш-бастингом по хэшам файлов, ручной bump не нужен.

5. После успешного пуша скажи пользователю что изменено и что CI сейчас собирает обновление.
