# Планирование следующего этапа — домашняя программа КПС

Это ретроспективный + проектный скилл: смотришь на **весь** пройденный отрезок (а не последние записи, как `analyze-results`) и предлагаешь **новый блок программы** — следующую фазу после текущей, или продолжение после дня 90. Точечные правки внутри уже написанных дней — это `adjust-program`. Содержание конкретных упражнений — по правилам `exercise-review`.

Если вердикт 🔴 — дождись правки `CLAUDE.md`, прежде чем продолжать. Советы из вердикта используй в разделе 4 при формулировке вопросов и в разделе 5 при проектировании этапа.

Прочитай `CLAUDE.md` и память — **ограничение по колену снимается только по факту осмотра ортопеда, не по времени**; не полагайся на то, что помнишь дату диагноза, проверь актуальный статус в проекте.

## 2. Проанализировать весь пройденный отрезок (не тренд последних записей — динамику целиком)

Для каждого показателя построй **траекторию от начала к концу отрезка**, а не только последние точки:

- **Поясница утром (`back_pain`)** — с какого значения стартовали, к какому пришли, были ли откаты (когда, после чего)
- **Левый КПС (`kps_feeling`)** — скорость роста 1→3, вышел ли на плато, на каком уровне
- **Левое колено (`left_knee`)** — стабильно ли 0, были ли обострения и в связи с чем (сверь даты с изменениями в `days.ts`, если есть история коррекций)
- **Симметрия при сидении (`sitting_symmetry`)** — дошли ли до «примерно ровно», сколько записей на это ушло

Прочитай прошлые `analysis_reports.content` целиком, а не только последний — это готовая история решений и того, что уже пробовали и почему (не повторяй уже проваленные подходы без причины).

## 3. Определить готовность к следующему этапу

Синтезируй по итогам раздела 2: какая фаза заканчивается, что в ней закрыто полностью, что закрыто частично, что не сдвинулось. Это база для решения, каким должен быть следующий этап — но **не решай в одиночку** то, чего не видно в данных: переходи к разделу 4.

## 4. Задать пользователю правильные вопросы — обязательный шаг

Данные из Supabase показывают только то, что попало в анкету. Прежде чем предлагать содержание следующего этапа, задай вопросы через `AskUserQuestion` **одним вызовом**, с конкретными вариантами (не открытые «расскажи как дела») — по темам, которые данные не покрывают:

- **Статус колена у ортопеда** — снят ли диагноз/ограничение с последнего осмотра, назначен ли новый осмотр. Это шлюзовой вопрос: ответ определяет, можно ли возвращать осевую нагрузку (присед, выпады, RDL на одной ноге) в программу.
- **Новые симптомы или травмы**, не попадавшие в ежедневную анкету (анкета покрывает только 4 показателя — спроси, не появилось ли что-то новое: плечо, шея, другое колено).
- **Изменение условий**: та же сидячая работа, то же время на тренировку (~60 мин), тот же набор гантелей дома, или что-то изменилось.
- **Приоритет следующего этапа**: дать варианты, а не спрашивать абстрактно — например: (а) продолжать текущий баланс таз+верх+кор, (б) сместить акцент на силовую прогрессию верха тела, (в) добавить больше кор-работы, (г) сосредоточиться на том, что осталось незакрытым по разделу 3.
- Если раздел 2 показал плато или откат по конкретному показателю — спроси прямой вопрос про этот конкретный показатель (например, откат по `back_pain` — спроси, был ли он привязан к конкретному дню/нагрузке, которую пользователь помнит, а данные не фиксируют).

Формулируй вопросы с готовыми вариантами ответа (`options`) и один вариант помечай как рекомендуемый, если у тебя есть основание из данных — но оставляй пользователю решение. Не задавай вопрос, ответ на который уже есть в данных или в `CLAUDE.md`.

## 5. Предложить план следующего этапа — сначала план, потом код

Собери синтез (раздел 3) + ответы пользователя (раздел 4) в текстовое предложение:

```
# Предложение: [следующая фаза] · [дата]

## Что показал пройденный отрезок
[3-5 предложений по разделу 2-3]

## Ответы, повлиявшие на план
[кратко, что учтено из раздела 4]

## Предлагаемая структура следующего этапа
- Недели: [диапазон]
- Фокус: [1-2 предложения]
- Что добавляется / что убирается из ротации / что продолжается
- Прогрессия силовых/удержаний относительно текущего уровня
```

Покажи это пользователю **как предложение** и дождись подтверждения — это новый многонедельный блок, а не точечная правка, ошибка здесь дороже, чем в `adjust-program`.

## 6. После подтверждения — реализация

1. Если нужны новые упражнения — следуй `exercise-review` (модель данных, `cues`, `note`, чек-лист ограничений) при добавлении в `src/data/exercises.ts`
2. Добавь новую фазу в `src/data/phases.ts` (следующий номер, диапазон недель, `icon`, `desc` в духе существующих)
3. Допиши дни в `src/data/days.ts`
4. Обнови версию service worker в `sw.js` (`kps-v` +1) — обязательно при пуше изменений в PWA
5. Закоммить и запушить:

```bash
cd /путь/к/проекту
git add src/data/phases.ts src/data/days.ts src/data/exercises.ts sw.js
git commit -m "phase: [краткое описание нового этапа и почему именно такой]"
git push origin main
```

6. Доложи пользователю: что показал анализ, что решили вопросы, что добавлено в программу и почему.
